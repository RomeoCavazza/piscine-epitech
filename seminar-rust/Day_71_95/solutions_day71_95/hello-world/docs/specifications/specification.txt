hello-world-rtc — Technical Specification & Logic Dictionary (UPDATED)
================================================================================
Project: Full-Stack Real-Time Messaging System
Tech Stack: Rust (Axum) + Next.js (React) + PostgreSQL + MongoDB

--------------------------------------------------------------------------------
1. CORE ARCHITECTURE
--------------------------------------------------------------------------------
- Clean Architecture implementation: Handlers (API) -> Services (Logic) -> Repositories (Data).
- Separation of Concerns: 
    - PostgreSQL: Strict relational data (Users, Servers, RBAC, Invites, DMs meta).
    - MongoDB: High-volume unstructured data (Channel/DM message history, Reactions).
- Auth: JWT based. Claims: sub (UUID), email (String), exp, iat.

--------------------------------------------------------------------------------
2. DATABASE MODELS (PostgreSQL)
--------------------------------------------------------------------------------
- users: id (UUID), username, email, password_hash, status (enum), avatar_url.
- servers: id, name, owner_id (FK -> users).
- server_members: server_id, user_id, role (OWNER/ADMIN/MEMBER).
- server_bans: server_id, user_id, banned_by, reason, expires_at (NULL = Permanent).
- channels: id, server_id, name, position.
- invites: id, code (Unique), server_id, created_by, max_uses, uses, revoked (Bool).
- direct_messages: id, user1_id, user2_id (Ordering: user1 < user2).

--------------------------------------------------------------------------------
3. DATABASE MODELS (MongoDB)
--------------------------------------------------------------------------------
- channel_messages: 
    - Fields: message_id (UUID), channel_id, server_id, author_id, content, reactions (List), created_at, edited_at, deleted_at (Soft Delete).
- dm_messages:
    - Fields: message_id (UUID), dm_id, author_id, content, reactions, created_at, deleted_at.

--------------------------------------------------------------------------------
4. WEBSOCKET PROTOCOL
--------------------------------------------------------------------------------
- Structure: { "op": "EVENT_NAME", "d": { "payload_data" } }
- Handshake:
    - Server -> HELLO { heartbeat_interval } immediately after upgrade on /ws.
    - Client -> IDENTIFY { token } as first logical auth step.
    - Server -> READY { user_id, username } on success, or ERROR on failure.
- Subscription model:
    - Client -> SUBSCRIBE / UNSUBSCRIBE by channel_id.
    - Channel broadcasts are sent only to subscribed channel sockets.
    - Direct-message events are pushed to all active sockets of both DM participants.
- Key Server Events:
    - MESSAGE_CREATE / MESSAGE_UPDATE / MESSAGE_DELETE / MESSAGE_REACTION_UPDATE.
    - DIRECT_MESSAGE_CREATE / DIRECT_MESSAGE_UPDATE / DIRECT_MESSAGE_DELETE / DIRECT_MESSAGE_REACTION_UPDATE.
    - PRESENCE_UPDATE with status enum: online | offline | dnd | invisible.
    - TYPING_START / TYPING_STOP for transient per-channel typing state.
    - HEARTBEAT_ACK, SUBSCRIBED, UNSUBSCRIBED, ERROR.
- Key Client Operations:
    - IDENTIFY: Handshake with JWT sent in payload, not query string.
    - SEND_MESSAGE: Persist and broadcast a channel message.
    - SUBSCRIBE: Join a channel room after auth + membership check.
    - TYPING_START / TYPING_STOP: Transient channel signals.
    - PRESENCE_UPDATE: Manual status update.
    - HEARTBEAT: Application-level heartbeat acknowledged by HEARTBEAT_ACK.

--------------------------------------------------------------------------------
5. BUSINESS LOGIC RULES
--------------------------------------------------------------------------------
- RBAC: 
    - OWNER: Full control, cannot leave (must transfer first).
    - ADMIN: Manage channels, kick/ban members, delete any message.
    - MEMBER: Chat, join/leave, delete own messages.
- Cross-DB Consistency: Resolved via business logic layer; no cross-DB transactions.
- Soft Delete: Messages are flagged with `deleted_at` across all store types.
- **Tauri Readiness**: Frontend is a pure static export (No Next.js server runtime required).
- **Internationalization**: Fully localized (FR/EN) with flag-based toggle on the client.
